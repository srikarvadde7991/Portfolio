SOC 2 Type I vs Type II Comparison

SOC 2 Type I

Focus: Design of controls at a point in time
Duration: Single assessment period (typically 1-3 months)
Scope: Control design, existence, and implementation
Testing: Limited operational testing (1-2 months)
Use: Vendor due diligence, initial assessment
Validity: 6-12 months (varies by use case)

SOC 2 Type II

Focus: Operational effectiveness over time
Duration: Extended period (minimum 6-12 months)
Scope: Control design, implementation, AND operating effectiveness
Testing: Extensive operational testing throughout period
Use: Major deals, customer trust, regulatory requirements
Validity: 1 year (requires renewal)

🔒 Encryption Policy

DATA ENCRYPTION POLICY

1. DATA AT REST ENCRYPTION
   - Production databases: AES-256
   - File systems: AES-256 (BitLocker/LUKS)
   - Backups: AES-256
   - Minimum key length: 256 bits
   - Key rotation: Annual or per incident

2. DATA IN TRANSIT ENCRYPTION
   - HTTPS/TLS 1.2+ for all connections
   - Certificate validation required
   - Strong cipher suites only
   - PFS (Perfect Forward Secrecy) enabled
   - No SSL 3.0 or TLS 1.0/1.1

3. KEY MANAGEMENT
   - Hardware Security Module (HSM) storage
   - Key escrow procedures
   - Access restricted to authorized personnel
   - Regular key audits
   - Documented key lifecycle

4. ENCRYPTION EXCEPTION PROCESS
   - Must be approved by CISO
   - Documented risk assessment
   - Compensating controls required
   - Annual re-assessment

Common Vulnerability Examples & Exploitation

SQLi

SQL Injection

Vulnerability Type: Injection Attack (OWASP A03)

Root Cause: Unsanitized user input concatenated into SQL queries

Exploitation: Craft malicious SQL to extract database records, modify data, or execute system commands

Business Impact: Complete database breach, unauthorized data access, compliance violations (PCI-DSS, HIPAA)

Tools Used: Burp Suite, SQLmap, Manual SQL crafting

XSS

Cross-Site Scripting

Vulnerability Type: XSS (OWASP A03)

Root Cause: Unescaped user input rendered in HTML/JavaScript context

Exploitation: Inject JavaScript to steal session cookies, perform phishing, deface pages

Business Impact: User credential theft, session hijacking, malware distribution

Tools Used: Burp Suite Repeater, Browser DevTools, Custom payloads

CSRF

Cross-Site Request Forgery

Vulnerability Type: CSRF (OWASP A01)

Root Cause: Missing/weak CSRF token validation on state-changing operations

Exploitation: Trick authenticated user into performing unwanted actions via malicious page

Business Impact: Unauthorized fund transfers, account modifications, data manipulation

Tools Used: Burp Suite, Custom HTML forms, Browser requests

IDOR

Insecure Direct Object Reference

Vulnerability Type: Broken Access Control (OWASP A01)

Root Cause: Missing or insufficient authorization checks on resource IDs

Exploitation: Modify resource IDs in requests to access other users' data

Business Impact: Unauthorized access to sensitive user data, privacy violations

Tools Used: Burp Suite Intruder, Repeater, Manual ID manipulation

Auth

Authentication Bypass

Vulnerability Type: Broken Authentication (OWASP A04)

Root Cause: Weak authentication logic, missing input validation, logic flaws

Exploitation: Bypass login mechanisms through parameter manipulation, JWT weaknesses, session fixation

Business Impact: Unauthorized account access, privilege escalation, account takeover

Tools Used: Burp Suite, Custom scripts, Kali tools (hydra, john)

API

API Security Vulnerabilities

Vulnerability Type: Insecure Design (OWASP A04)

Root Cause: Missing rate limiting, weak authentication, improper endpoint authorization

Exploitation: API enumeration, unauthorized data access, brute force attacks, privilege escalation

Business Impact: Massive data exposure, service disruption, unauthorized actions

Tools Used: Burp Suite, Postman, Custom API clients, curl

# IDOR exploitation - access other users

            'proof_of_concept': vuln['poc_url']
        })

rule detect_credential_stuffing {
  meta:
    author = "Security Team"
    severity = "HIGH"
    mitre_tactic = "Credential Access"
  events:
    $failed = authentication 
      where event.status == "FAILED"
      and count() >= 10
    $success = authentication 
      where event.status == "SUCCESS"
  condition:
    $failed followed by $success 
    within 5m
}
